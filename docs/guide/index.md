# Forge 接入总览

AsterForge 的目标不是把产品项目改造成一个框架，而是把多个 Aster 服务里已经重复出现的基础设施抽出来，形成可复用、可测试、边界清楚的 Rust crates。

接入时先判断代码属于哪一层：

- **产品语义**：用户、团队、权限、业务状态机、数据库实体、配置存储方式，留在产品仓库。
- **共享机制**：分页、缓存、数据库连接、指标接口、任务租约、重试分类、路径和 URL 规范化，放在 Forge。
- **适配边界**：把 Forge 的通用错误、trait、配置结构映射到产品错误、产品指标和产品 repository。

## 推荐接入顺序

1. 从最小 crate 开始，例如 `aster_forge_validation`、`aster_forge_utils`、`aster_forge_api`。
2. 再接入生命周期类 crate，例如 `aster_forge_logging`、`aster_forge_metrics`、`aster_forge_panic`。
3. 最后接入会影响业务流程的 crate，例如 `aster_forge_cache`、`aster_forge_db`、`aster_forge_external_auth`、`aster_forge_tasks`。

这样做的原因很简单：小工具 crate 容易验证一致性，生命周期和任务系统一旦接入会影响启动、关闭、错误处理、测试隔离，不能乱来。

## 文档结构

每个 crate 页面都按同一套结构写：

- **用途边界**：这个 crate 解决什么，不解决什么。
- **功能开关**：Cargo feature 应该如何启用。
- **最小接入**：产品仓库里通常应该怎么调用。
- **错误边界**：Forge 错误应该在哪里映射成产品错误。
- **测试要求**：接入时必须覆盖哪些行为。
- **参考项目**：Drive/Yggdrasil 里可以对照的代码路径。

## 当前设计取舍

Forge 默认不做下面这些事：

- 不持有产品数据库实体。
- 不定义产品级权限模型。
- 不重新导出产品仓库的类型。
- 不替产品仓库决定用户可见错误文案。
- 不隐藏需要调用方显式处理的外部系统失败。

如果一个功能看起来“可以抽”，但抽出来以后会让产品仓库无法表达自己的业务边界，那就先别抽。猫猫别把所有东西都塞进共享库，塞进去容易，拆出来才烦。
